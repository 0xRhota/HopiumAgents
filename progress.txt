## Codebase Patterns

- Grid MM scripts are in `scripts/` (grid_mm_hibachi.py, grid_mm_nado_v8.py, grid_mm_live.py)
- Hibachi uses 30s time-based refresh (working correctly)
- Nado/Paradex use price/fill-based refresh (broken - orders go stale)
- Dynamic spread is calculated in `_calculate_dynamic_spread()` method
- ROC threshold for pause is configurable via `roc_threshold_bps` parameter
- All bots use POST_ONLY orders to guarantee maker fills
- Exchange SDKs are in `dexes/` directory
- Tests for dynamic spread logic: `tests/test_dynamic_spread.py`

## 2026-01-22 - Initial State Assessment

### Issues Identified:

1. **Nado Grid MM**: 0 fills in 3.5+ hours
   - 102% inventory (LONG), can only SELL
   - Dynamic spread working (30bps at high ROC)
   - PAUSE logic working (pauses during trends)
   - Problem: No buyers on Nado DEX - liquidity issue
   - File: `scripts/grid_mm_nado_v8.py`

2. **Paradex Grid MM**: 0 fills in 1+ hour
   - Stale orders from 08:10 sitting $70+ from market
   - 0.5% price reset threshold too conservative
   - "No active orders" check doesn't help (stale orders still OPEN on exchange)
   - File: `scripts/grid_mm_live.py`

3. **Both bots**: Spread doesn't update without order placement
   - `_calculate_dynamic_spread()` only called inside `_place_grid_orders()`
   - With 0 fills and <0.5% price move, spread never recalculates
   - Evidence: ROC at 27.8bps but spread stuck at 15bps

4. **Hibachi Grid MM**: Working correctly
   - 30s time-based refresh prevents stale orders
   - Dynamic spread updates every cycle
   - This is the model to follow

### Learnings for future iterations:

- Time-based refresh (like Hibachi) is essential for low-liquidity markets
- Price-based refresh (0.5% threshold) is too conservative
- DEX liquidity varies significantly - Nado may not be viable for grid MM
- Always check if orders are actually being placed, not just tracked
